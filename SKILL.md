# 盘古 · 开天术

> 「知识库是图书馆，Skill 是专家」——把你的知识库从「被动查询」变成「主动能力」。

> 本 Skill 借鉴了 [女娲 · Skill](https://github.com/alchaincyf/nuwa-skill) 的成熟经验，采用多阶段流程 + 检查点设计 + 质量验证标准，非常感谢。

---

## 核心理念

| 维度 | 知识库 | Skill |
|------|--------|-------|
| 使用方式 | 搜索 → 读取 → 理解 | 触发关键词 → 自动执行 |
| 知识形态 | 原始文档（散装） | 结构化流程 + 精炼规则 |
| 门槛 | 需要知道搜什么、怎么问 | 说一句话就触发完整流程 |
| 复用性 | 每次都要搜+读+理解 | 安装即用，开箱即得 |
| 分发 | 加入知识库 → 问答 | 安装 Skill → 直接用 |

**关键区分**：蒸馏的是**能做到什么**（流程/规则/框架），不是**知道什么**（信息/文档）。

---

## 执行流程

### Phase 0: 入口分流

收到用户输入后，先判断蒸馏模式：

| 模式 | 触发特征 | 示例 |
|------|---------|------|
| **完整蒸馏** | 提到知识库名称，未指定范围 | 「把 @XX知识库 蒸馏成 Skill」 |
| **文件夹蒸馏** | 指定了文件夹名称 | 「只蒸馏「XX文件夹」里的内容」 |
| **单文档蒸馏** | 指定了具体文档 | 「把「XX文档」蒸馏成 Skill」 |
| **任务蒸馏** | 提到「刚才的任务」「工作流」 | 「把我刚才做的调研报告流程变成 Skill」 |

---

### Phase 1: 需求澄清

#### 1.1 确认基本信息

根据蒸馏模式，确认关键信息：

**完整蒸馏模式**：
```
请确认以下信息：

【基本信息】
知识库：@{知识库名称}
这个知识库主要解决什么问题：{一句话描述}
希望别人安装这个 Skill 后能做什么：{一句话描述}

【技能定制】（可选）
技能名称：{英文短名，如 contract-review}（不填自动生成）
触发关键词：{逗号分隔}（不填自动推断）
不适用场景：{不应该触发的场景}（不填自动推断）

【内容范围】（可选）
重点提炼：{必须包含的内容}
编程规范/易错点：{重要的注意事项、规则}
参考文档：{是否需要保留详细参考文档}
```

**文件夹蒸馏模式**：
```
请确认以下信息：

知识库：@{知识库名称}
蒸馏范围：只提炼「{文件夹A}」和「{文件夹B}」文件夹里的内容
这个知识库主要解决什么问题：{一句话描述}
希望别人安装这个 Skill 后能做什么：{一句话描述}
```

**单文档蒸馏模式**：
```
请确认以下信息：

知识库：@{知识库名称}
蒸馏范围：只提炼「{文档名称}」
这个文档主要解决什么问题：{一句话描述}
希望别人安装这个 Skill 后能做什么：{一句话描述}
```

**任务蒸馏模式**：
```
请确认以下信息：

刚才完成的工作：{简要描述}
这个 Skill 要帮别人做什么：{一句话描述}
触发关键词：{别人说什么话应该触发}
```

#### 1.2 检查知识库是否适合蒸馏

在开始之前，评估知识库是否适合蒸馏：

| ✅ 适合蒸馏 | ❌ 不适合蒸馏 |
|------------|-------------|
| 有明确流程/步骤 | 纯数据型（价格表、产品目录） |
| 有规则/规范检查 | 需要频繁更新的实时数据 |
| 有分类决策场景 | 纯文献存档（无操作指导） |
| 有方法论/框架 | 一次性查阅的参考资料 |
| 有固定输出格式 | 需要专业资质的领域（医疗诊断等） |

**如果不适合**：在 Phase 1 告知用户，并建议更适合的方案。

用户说「就做XX」没有更多信息 → 默认全面蒸馏，直接推进。
用户提供了详细信息 → 标记为**精细模式**，后续检查点更严格。

确认后 → 跳到 Phase 1.5。

---

### Phase 1.5: 创建 Skill 目录

**收到确认后立即执行**，在读取内容之前完成：

```
{skill-name}/
├── SKILL.md                          # 核心技能文件
├── references/                       # 参考文档（从知识库提炼）
│   ├── 01-core-knowledge.md         # 核心知识提炼
│   ├── 02-checklist.md             # 检查清单（如有）
│   └── 03-cases.md                 # 案例库（如有）
├── assets/                          # 模板文件（从知识库提取）
│   └── {template}.md
└── examples/                        # 使用示例
    └── {example}.md
```

**完成检查**（自动执行）：
- [ ] 目录已创建
- [ ] 如果是更新模式：已读取现有 SKILL.md，标注哪些信息需要刷新
- [ ] 如果知识库很大（>100文件）：已建议用户指定蒸馏范围

**关键规则**：
- 所有参考文件必须存在 skill 目录内部（`references/`），绝对不要存到外部目录。
- Skill 必须是自包含的——复制整个 skill 目录就能独立使用。

---

### Phase 2: 内容读取与分析

#### 2.1 读取知识库内容

根据用户提供的知识库信息，使用 `@知识库名称` 引用方式读取内容。

**读取策略**：

| 蒸馏模式 | 读取策略 |
|---------|---------|
| 完整蒸馏 | 读取知识库所有文件和文件夹结构 |
| 文件夹蒸馏 | 只读取指定文件夹内的文件 |
| 单文档蒸馏 | 只读取指定的单篇文档 |
| 任务蒸馏 | 分析对话历史，提取完整工作流 |

#### 2.2 内容结构化分析

对读取的内容进行结构化分析：

```
【核心知识提炼】（必须）
- 关键概念及定义（3-5个）
- 核心方法论/框架（1-3个）
- 重要原则和规则（5-10条）

【流程提取】（如果有）
- 操作步骤（按顺序列出）
- 决策流程（如有分支）
- 检查清单（如有）

【案例提取】（如果有）
- 典型案例（2-5个）
- 常见问题（3-5个）
- 最佳实践（3-5个）

【参考文档识别】
- 需要完整保留的参考文档
- 需要精简的规则文档
- 可以作为模板的文件
```

#### 2.3 知识类型判断

根据知识库内容判断应该蒸馏成哪种类型的 Skill：

| 知识类型 | Skill 类型 | 结构特点 |
|---------|-----------|---------|
| 有明确流程/步骤 | 流程型 Skill | Phase/Step 步骤 + 决策点 |
| 有规则/规范检查 | 规范型 Skill | 检查清单 + 判断标准 |
| 有分类决策场景 | 任务型 Skill | 决策表 + 场景路由 |
| 有方法论/框架 | 框架型 Skill | 核心框架 + 应用指南 |
| 有固定输出格式 | 模板型 Skill | 模板 + 填充规则 |

---

### Phase 2.5: 分析确认检查点

**内容读取和分析完成后，暂停展示分析摘要**：

```
┌──────────────────┬────────────┬──────────────────────────────┐
│ 分析维度         │ 数量       │ 关键发现                      │
├──────────────────┼────────────┼──────────────────────────────┤
│ 核心概念         │ N个        │ {概念1}、{概念2}...           │
│ 流程步骤         │ N步        │ {步骤1}→{步骤2}→...          │
│ 规则/规范        │ N条        │ {规则1}、{规则2}...          │
│ 典型案例         │ N个        │ {案例1}、{案例2}...          │
│ 模板/文档        │ N个        │ {模板1}、{模板2}...          │
├──────────────────┼────────────┼──────────────────────────────┤
│ 建议的Skill类型  │ {类型}     │ {判断理由}                    │
│ 预计难度         │ 低/中/高   │ {原因}                        │
└──────────────────┴────────────┴──────────────────────────────┘
```

用户确认分析结果 OK → 进入 Phase 3。
用户觉得分析有遗漏或偏差 → 补充读取后再继续。

**这个检查点的意义**：分析质量决定了最终 Skill 的上限。垃圾进垃圾出，在这里拦截比在 Phase 4 返工成本低得多。

---

### Phase 3: 框架提炼

#### 3.1 核心框架提取（3-7个）

**操作步骤**：

1. **扫描**：列出所有候选知识（反复出现的规则、自创术语、核心主张）。通常会得到 10-30 个候选
2. **筛选**：按重要性排序，取 top 3-7 个。宁少勿多——3 个核心框架远好于 10 个浅薄的原则
3. **记录格式**：每个框架记录——名称、一句话描述、来源证据（≥2 个场景）、应用方式、局限性

#### 3.2 流程步骤提炼（如有）

= 完成某项任务的具体步骤。可表述为「先 X，再 Y，最后 Z」，有具体操作指引。

#### 3.3 检查清单提炼（如有）

= 需要逐项确认的内容。可表述为「在做 X 之前，确认以下 N 项」。

#### 3.4 决策规则提炼（如有）

= 做判断时的快速规则。可表述为「如果 X，则 Y」，有具体案例支撑。

#### 3.5 模板提炼（如有）

= 需要生成特定格式内容时的模板。

#### 3.6 边界规则

必须明确写出的局限：
- 这个 Skill 不能做什么
- 在什么情况下不适用
- 需要人工介入的场景

---

### Phase 3.5: 提炼确认检查点

Phase 3 提炼完成后，暂停展示提炼摘要给用户确认：

```
提炼结果摘要：
- 核心框架：N个（列出名称）
- 流程步骤：N步
- 检查清单：N项
- 决策规则：N条
- 输出模板：N个
- 边界规则：N条
```

用户确认 OK → 进入 Phase 4 构建。
用户觉得某个框架不对或缺少 → 回到 Phase 3 调整后再继续。

**这个检查点的意义**：提炼是主观判断最重的环节，确认后再构建，避免写完 SKILL.md 才发现方向不对。

---

#### 4.2 内容映射规则

| 知识库内容 | 映射为 Skill 中的 |
|-----------|------------------|
| 零散的规则/规范 | 规范型 Skill 的检查清单 |
| 操作步骤文档 | 流程型 Skill 的 Step 步骤 |
| 分类知识（不同场景不同做法） | 任务型 Skill 的决策表 |
| 长篇参考文档 | references/ 目录下的参考文件 |
| 模板/范例 | assets/ 目录下的模板文件 |
| 案例/常见问题 | examples/ 目录下的示例文件 |

#### 4.3 构建完成检查

构建完成后，检查以下事项：

- [ ] SKILL.md 包含完整的 frontmatter（name, description, trigger）
- [ ] SKILL.md 包含适用/不适用场景说明
- [ ] SKILL.md 包含执行流程
- [ ] 核心框架有来源证据和应用方式
- [ ] 边界规则明确写出局限
- [ ] 参考文档已放入 references/ 目录
- [ ] 模板文件已放入 assets/ 目录

---

### Phase 5: 质量验证

生成 Skill 后，执行以下质量验证：

#### 5.1 已知测试（Sanity Check）

选 3 个知识库中明确提到过的场景，用新生成的 Skill 回答：
- 回答正确 → 框架有效
- 回答错误 → 回溯调整框架

#### 5.2 边缘测试（Edge Case）

选 1-2 个知识库中没明确提到但相关的问题，用 Skill 推断：
- 期望结果：「基于框架 X 和规则 Y 的推断，可能...但不确定」
- 不应该斩钉截铁

#### 5.3 可用性测试

用 Skill 写一段使用说明，判断：
- 流程是否清晰可执行？
- 示例是否足够帮助理解？
- 边界是否明确？

#### 5.4 通过标准

| 检查项 | 通过标准 | 不通过信号 |
|--------|---------|-----------|
| 核心框架数量 | 3-7 个，每个有来源证据 | <3 或 >10 |
| 每个框架的局限性 | 明确写出失效条件 | 只写优点 |
| 流程步骤清晰度 | 用户能按步骤执行 | 步骤模糊或缺失 |
| 边界规则完整性 | 至少 2 条具体局限 | 只有「不能替代人工」 |
| 示例充分性 | 至少 2 个具体示例 | 没有示例 |
| 文档结构完整性 | 所有必要 section 都有 | 有 section 缺失 |

验证通过 → 进入 Phase 6。
不通过 → 标注薄弱环节，回到 Phase 3 迭代。

**迭代上限**：Phase 3→5 最多循环 2 次。如果 2 轮后仍有不通过项，在边界规则中标注薄弱维度，交付当前最优版本而非无限打磨。

**展示验证结果给用户确认后才算完成。**

---

### Phase 6: 双Agent精炼（可选）

Phase 5 验证通过后，可选启动双Agent精炼：

**Agent A（结构视角）**：
- 评审 SKILL.md 的结构清晰度
- 评审工作流是否「激活即执行」
- 输出：最弱 2 个维度的具体改进建议

**Agent B（用户视角）**：
- 评审触发关键词是否能准确触发
- 评审示例是否足够帮助理解
- 输出：2-3 处具体文本改动建议

**主Agent综合两份报告，应用不冲突的改进，展示变更摘要请用户确认。**

精炼标准：改动必须让 skill「激活即执行」，不只是增加内容，而是让 AI 拿到 skill 后知道先做什么、碰到什么停下来。

---

### Phase 7: 输出与注册

#### 7.1 输出 Skill 文件

将生成的 Skill 文件输出到用户指定的目录，或提供下载链接。

#### 7.2 注册指引

```
要在 IMA 中注册这个 Skill：

1. 将 Skill 文件夹打包为 zip
2. 在 IMA 的 Skill 管理页面上传
3. 等待审核通过（通常需要 1-2 个工作日）
4. 审核通过后，在新的对话中即可触发使用
```

## 更新已有 Skill

当用户说「更新我的 XX Skill」「XX 最近有新内容」时：

1. 读取现有的 SKILL.md，从「注意事项」section 中找到「更新时间：[日期]」，标注距今多久
2. 只读取变化的部分（新增/修改的文档）
3. 对比新内容与现有内容：
   - 新内容强化现有框架 → 补充案例
   - 新内容与现有框架矛盾 → 标注变化，更新框架
   - 出现新的方法论 → 考虑增加新框架
4. 更新 SKILL.md 中的更新时间
5. 不重写整个 Skill，只增量更新

## 品味守则（速查）

遇到判断困难时回看。

| 原则 | 一句话 |
|------|--------|
| 流程 > 知识 | 能执行的流程比散装知识更有价值 |
| 规则 > 案例 | 具体规则比一堆案例更容易泛化 |
| 边界 > 能力 | 明确边界比夸大能力更诚实 |
| 精炼 > 堆砌 | 3 个核心框架远好于 10 个浅薄原则 |

### 绝不做的事

- 把知识库文档原封不动地复制到 Skill 中
- 蒸馏不适合蒸馏的知识库类型
- 夸大 Skill 的能力（超出知识库实际覆盖范围）
- 在信息不足时强行生成完整的 Skill

## 特殊场景

### 场景1：知识库内容太多

如果知识库内容超过 100 个文件或 50000 字：

1. 建议用户指定蒸馏范围（文件夹或文档）
2. 或者分批次蒸馏，先蒸馏核心部分
3. 在 Phase 1 就提醒用户：「这个知识库内容较多，建议指定蒸馏范围以提高质量」

### 场景2：知识库不适合蒸馏

以下类型的知识库不适合蒸馏为 Skill：

- 纯数据型（如价格表、产品目录）→ 建议使用知识库 + 搜索
- 需要频繁更新的实时数据 → 建议使用知识库 + 定期更新
- 纯文献存档（无操作指导价值）→ 建议使用知识库
- 需要专业资质的领域（医疗诊断、法律咨询等）→ 建议保留为参考，人工决策

如果遇到这类知识库，在 Phase 1 就告知用户，并建议更适合的方案。

### 场景3：跨文件夹引用

如果知识库中的文档有相互引用：

1. 在蒸馏时保留引用关系
2. 在 SKILL.md 中使用相对路径引用 references/ 下的文件
3. 确保所有被引用的内容都被蒸馏到 Skill 中

### 场景4：蒸馏用户自己的知识库

当用户说「把我的知识库蒸馏成 Skill」「我想把自己的经验变成 Skill」时：

1. 需要用户明确授权和描述核心知识
2. 引导用户提供：核心方法论、重要规则、典型案例、自我总结
3. 注意「自我认知偏差」——用户可能高估某些特质、忽略盲点
4. 可以追问：「你觉得最核心的规则是什么？」「有什么常见的错误？」

### 场景5：冷门知识库（公开信息极少）

当知识库内容非常有限（<10 篇文档）时：

1. 在 Phase 1.5 就告知用户「这个知识库内容较少，生成的 Skill 质量会受限」
2. 框架数量减至 2-3 个
3. 边界规则 section 加大篇幅，明确列出「哪些维度信息不足」
4. 建议用户提供更多素材或案例

## 小贴士

| 问题 | 建议 |
|------|------|
| 不知道技能名称怎么起？ | 不填，让 AI 自动生成（英文 kebab-case 格式） |
| 不知道触发关键词怎么写？ | 不填，AI 会根据知识库内容自动推断 |
| 知识库内容太多？ | 在"重点提炼"中说明哪些必须包含、哪些可以忽略 |
| 只想蒸馏部分内容？ | 加一行「蒸馏范围：只提炼「XX」文件夹里的内容」|
| 想修改已蒸馏的 Skill？ | 说"更新我的 XX 技能"或"修改我的 XX 技能" |
| 想分享给别人？ | 告诉 AI "我需要下载 zip 包" |

## 注意事项

1. **知识库必须可访问** — 使用 `@知识库名称` 引用，确保 AI 能读取内容
2. **不是所有知识库都适合蒸馏** — 纯数据型知识库（如价格表）不如有方法论/流程的知识库效果好
3. **蒸馏后知识库仍保留** — Skill 是知识库的精炼版，不会删除原知识库
4. **需要审核** — 注册后需等平台审核通过才可使用
5. **新对话生效** — 审核通过后，在新的对话中才能触发该 Skill
6. **内容版权** — 确保你有权蒸馏该知识库的内容

## 最后

这个 Skill 的目标是：让你知识库中的知识，从「被动等待查询」变成「主动帮你工作」。

一个好的蒸馏 Skill，应该让用户在安装后，只需要说一句话，就能触发完整的工作流——就像有一个专家在旁边指导一样。

**蒸馏的是能做到什么（流程/规则/框架），不是知道什么（信息/文档）。**
