# 知识库蒸馏最佳实践

## 什么值得蒸馏？

### ✅ 适合蒸馏的知识库

| 特征 | 示例 | 蒸馏后的价值 |
|------|------|-------------|
| 有明确流程/步骤 | 合同审查流程、代码审查清单 | 用户说一句话就触发完整流程 |
| 有规则/规范检查 | API 规范、代码风格指南 | 自动检查并给出修改建议 |
| 有分类决策场景 | 不同客户类型的处理方案 | 根据用户输入自动路由到正确方案 |
| 有方法论/框架 | OKR 制定方法、用户访谈框架 | 引导用户按框架思考和执行 |
| 有固定输出格式 | 周报模板、需求文档模板 | 自动生成符合规范的文档 |

### ❌ 不适合蒸馏的知识库

| 特征 | 示例 | 建议方案 |
|------|------|---------|
| 纯数据型 | 价格表、产品目录、通讯录 | 保留为知识库，用搜索查询 |
| 需要频繁更新的实时数据 | 股价、天气、实时新闻 | 用 API 或实时数据源 |
| 纯文献存档 | 学术论文、法律条文原文 | 保留为知识库，用于查阅 |
| 一次性参考资料 | 某个具体项目的文档 | 保留为知识库，按需查询 |

## 蒸馏质量原则

### 1. 提炼，不要复制

❌ **错误做法**：把知识库的文档原封不动地复制到 Skill 中
✅ **正确做法**：提炼核心规则、流程、决策点

**示例**：

知识库原文（500字）：
> 合同审查需要注意很多方面。首先要看合同双方的基本信息是否完整，包括名称、地址、法定代表人等。然后要审查合同标的是否明确...

蒸馏后（50字）：
> **合同审查三步法**：①核对双方信息 → ②确认合同标的 → ③识别风险条款

### 2. 结构化，不要散装

❌ **错误做法**：把所有规则堆砌在一起
✅ **正确做法**：用表格、列表、流程图组织

**示例**：

散装（难用）：
> 如果合同金额大于10万需要法务总监审批，如果在国外需要合规审查，如果是采购合同需要采购部门确认...

结构化（好用）：
| 合同类型 | 金额阈值 | 审批人 |
|---------|---------|--------|
| 采购合同 | >10万 | 法务总监 |
| 任何合同 | 涉及国外 | 合规部门 |
| 任何合同 | 任何金额 | 采购部门（如适用）|

### 3. 场景化，不要抽象

❌ **错误做法**：写一堆抽象的原则
✅ **正确做法**：给出具体场景和示例

**示例**：

抽象（难理解）：
> 要注意合同条款的公平性，避免对客户产生不利影响的条款...

场景化（好理解）：
> ❌ **不公平条款示例**："甲方有权随时终止合同，无需说明理由"
> ✅ **修改后**："甲方终止合同需提前30天书面通知，并说明合理理由"

### 4. 保留边界，不要过度泛化

❌ **错误做法**：把特定场景的规则推广到所有场景
✅ **正确做法**：明确标注规则的适用边界

**示例**：

过度泛化（危险）：
> 所有合同都必须有违约金条款（❌ 实际上有些合同不需要）

保留边界（安全）：
> **需要违约金条款的合同**：服务合同、租赁合同、采购合同
> **可省略违约金条款的合同**：赠与合同、无偿保管合同

## 蒸馏步骤详解

### Step 1：浏览全貌（5分钟）

不要一上来就蒸馏，先花5分钟浏览知识库：

1. 看文件夹结构 → 理解知识组织方式
2. 看文档标题 → 识别核心主题
3. 看文档数量 → 判断是否需要指定蒸馏范围

**输出**：在心里形成一个知识地图

### Step 2：识别核心（10分钟）

问自己三个问题：

1. **用户用这个知识库最想解决什么问题？**（核心场景）
2. **解决这个问题最关键的知识是什么？**（核心内容）
3. **如果只能保留20%的内容，保留哪些？**（精炼核心）

**输出**：列出 3-5 个核心知识点

### Step 3：选择蒸馏类型（2分钟）

根据核心知识点的类型，选择 Skill 类型：

| 核心知识点类型 | Skill 类型 | 结构特点 |
|---------------|-----------|---------|
| 是一系列步骤 | 流程型 | Step 1 → Step 2 → Step 3 |
| 是一系列检查项 | 规范型 | 检查清单 + 通过标准 |
| 是不同场景的不同做法 | 任务型 | 决策表 + 场景路由 |
| 是一个思考框架 | 框架型 | 核心框架 + 应用指南 |

### Step 4：提炼和结构化（20-30分钟）

对每条核心知识，进行提炼：

1. **提取关键词**：这条知识的3-5个关键词是什么？
2. **简化为规则**：能否用一句话说完？
3. **示例化**：能否给出一个具体示例？
4. **边界化**：这条规则在什么情况下不适用？

**输出**：结构化的 Skill 内容

### Step 5：验证和优化（10分钟）

自己扮演用户，用这个 Skill：

1. 说一句触发语 → Skill 能正确触发吗？
2. 按 Skill 指引操作 → 能解决我的问题吗？
3. 遇到边界情况 → Skill 有应对吗？

**输出**：优化后的 Skill

## 常见蒸馏错误

### 错误1：试图蒸馏所有内容

**问题**：知识库有100篇文档，试图全部蒸馏到 Skill 中
**后果**：Skill 变得庞大而难用，失去了「精炼」的意义
**解决**：只蒸馏核心的 20% 内容，其余 80% 作为参考文档保留

### 错误2：蒸馏后无法触发

**问题**：Skill 的 trigger 设置得太窄，用户很难触发
**后果**：用户安装了 Skill 却用不起来
**解决**：设置 5-10 个常见的触发关键词，覆盖用户的自然表达

### 错误3：流程步骤不清晰

**问题**：Skill 中的流程步骤模糊，用户不知道下一步该做什么
**后果**：用户中途放弃使用
**解决**：每个步骤都给出「输入是什么」「输出是什么」「怎么做」

### 错误4：缺少示例

**问题**：Skill 只有规则，没有示例
**后果**：用户不理解如何应用规则
**解决**：每个重要规则都配一个具体示例（最好有正反对比）

### 错误5：没有边界说明

**问题**：Skill 没有说明「什么情况下不适用」
**后果**：用户在不适用的场景使用，得到错误结果
**解决**：在 SKILL.md 中明确写出「不适用场景」

## 质量自检清单

蒸馏完成后，用这个清单检查：

### 结构完整性

- [ ] SKILL.md 有 frontmatter（name, description, trigger）
- [ ] 有适用场景说明（✅ 和 ❌）
- [ ] 有清晰的执行流程
- [ ] 有注意事项/边界说明

### 内容质量

- [ ] 触发关键词是否覆盖用户的自然表达？
- [ ] 流程步骤是否清晰可执行？
- [ ] 是否有具体示例？
- [ ] 是否有边界说明？

### 蒸馏充分性

- [ ] 是否保留了核心知识（20%）？
- [ ] 是否去除了无关内容（80%）？
- [ ] 结构是否比原知识库更清晰？
- [ ] 用户能看懂并使用吗？

## 进阶技巧

### 技巧1：用决策表替代长篇说明

如果有「不同情况不同做法」的知识，用决策表：

**长篇说明（难用）**：
> 如果是新客户，需要... 如果是老客户，需要... 如果是一级客户，需要... 如果是二级客户...

**决策表（好用）**：
| 客户类型 | 所需材料 | 审批流程 |
|---------|---------|---------|
| 新客户 | A+B+C | 三级审批 |
| 老客户 | B+C | 二级审批 |
| 一级客户 | C | 一级审批 |

### 技巧2：用检查清单替代描述性文字

如果有「需要确保多项内容」的知识，用检查清单：

**描述性文字（难用）**：
> 在发布前，你需要确保测试都通过，文档都更新，代码都review...

**检查清单（好用）**：
Release 检查清单：
- [ ] 所有测试通过
- [ ] 文档已更新
- [ ] Code Review 已完成
- [ ] 发布通知已发送

### 技巧3：用模板替代格式说明

如果有「需要生成特定格式内容」的知识，提供模板：

**格式说明（难用）**：
> 周报需要包含本周工作、下周计划、风险点三个部分...

**模板（好用）**：
```markdown
# 周报 - {姓名} - {日期}

## 本周工作
- 

## 下周计划
- 

## 风险点
- 
```

## 总结

好的蒸馏 = **20% 的核心内容 + 清晰的结构 + 具体的示例 + 明确的边界**

记住：**Skill 是专家，不是图书馆**。用户想要的是「专家帮我做」，而不是「专家给我一堆书让我自己读」。
